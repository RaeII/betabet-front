# 012 — PWA (Progressive Web App) instalável

Plano detalhado para transformar o **Bolão CLT** em um PWA instalável (Android,
iOS/iPadOS e desktop), com app shell offline, ícone na tela inicial, splash
screen e fluxo de atualização controlado — **sem alterar a stack atual nem o
modelo de autenticação por cookie**.

> Status deste documento: **implementado** (MVP instalável). As Fases 1–8 abaixo
> foram aplicadas; o build (`bun run build`) gera `dist/sw.js`,
> `dist/manifest.webmanifest` e os ícones, e o app atende aos critérios de
> instalabilidade do Chrome. Push real para mensagens do chat foi implementado
> depois em `017-push-notifications.md`.

---

## 1. Objetivo e escopo

### Objetivo

Permitir que o usuário **instale** o Bolão CLT na tela inicial do celular e
abra a aplicação em modo standalone (sem barra do navegador), com carregamento
instantâneo do app shell mesmo em rede ruim, e um fluxo claro de atualização
quando sair uma versão nova.

### Dentro do escopo (MVP instalável)

- `manifest.webmanifest` com nome, ícones, cores e `display: standalone`.
- Conjunto completo de ícones (incl. **maskable** e **apple-touch-icon**).
- Service Worker via `vite-plugin-pwa` (Workbox) com **precache do app shell**.
- Estratégia de cache segura para assets vs. API vs. sessão.
- Componente de UI para **prompt de atualização** ("Nova versão disponível").
- Meta tags específicas de iOS no `index.html`.
- Ajuste de headers de cache no nginx para `sw.js`/manifest.
- Critérios de aceite e roteiro de testes (Lighthouse + instalação real).

### Fora do escopo deste MVP instalável

- **Push notifications reais** ficaram fora deste MVP inicial. Hoje há Web Push
  para `chat.message` em `017-push-notifications.md`; notificações de partidas,
  resultados e palpites seguem futuras.
- **Background Sync** de palpites offline (enfileirar POST/PUT de aposta quando
  sem rede). Possível, mas requer cuidado com `usePlaceBet`/`useEditBet` e a
  aposta otimista — adiar.
- Cache offline de dados dinâmicos (jogos, ranking, palpites). O MVP mantém
  esses dados **sempre via rede**; offline mostra o app shell + estado vazio.

## 2. Diagnóstico do código atual

### 2.1 Stack relevante

| Item | Situação | Implicação para PWA |
| --- | --- | --- |
| Bundler | **Vite 6** (`vite.config.ts`) | `vite-plugin-pwa` é o caminho oficial e trivial |
| Framework | React 19 + React Router 7 | SPA com `createBrowserRouter` |
| Code splitting | **Já existe** (`lazy: async () => import(...)` em `src/router/index.tsx`) | Workbox precacheia só o shell; rotas viram cache sob demanda |
| Estado de servidor | TanStack Query (`src/main.tsx`) | Camada de cache em memória já isola dados dinâmicos |
| Cliente HTTP | `src/services/api.ts` com `credentials: 'include'` | **Auth por cookie httpOnly** → não cachear `/api/` |
| Build | Bun (`Dockerfile`, `bun run build`) | `vite build` já gera o SW junto |
| Deploy | nginx em container (`nginx/default.conf.template`) | Já faz `try_files ... /index.html` (SPA ok) |

### 2.2 O que **já** está pronto (favorável)

- `index.html` já tem `<meta name="theme-color" content="#123D2A">`,
  `description`, `application-name`, `lang="pt-BR"` e favicon.
- Logos disponíveis em `public/`: `bolao_clt_logo.png`, `bolao_clt_fav.png`,
  `bolao_clt_bg.png`, `favicon-64.png`.
- nginx serve a SPA corretamente: `location / { try_files $uri $uri/ /index.html; }`
  com `Cache-Control: no-cache`, e `/assets/` com `immutable` (hashes do Vite).
- Rotas já são lazy → o precache inicial fica pequeno (só shell + chunk de entrada).

### 2.3 O que **falta**

- Não há **nenhum** service worker, `manifest`, `vite-plugin-pwa` nem
  `registerSW` (confirmado por grep em `src/`, `public/` e `index.html`).
- Ícones quadrados em múltiplos tamanhos (192/512) e **maskable** — o
  `bolao_clt_logo.png` atual é 500×668 (**não é quadrado**), então precisa ser
  re-exportado/centralizado num canvas quadrado com padding seguro.
- Header de cache dedicado para `sw.js` e `manifest.webmanifest` no nginx.

## 3. Avaliação de dificuldade

**Nível geral: BAIXO–MÉDIO.** O projeto é quase o cenário ideal para PWA.

| Tarefa | Dificuldade | Observação |
| --- | --- | --- |
| Instalar/configurar `vite-plugin-pwa` | 🟢 Baixa | Plugin oficial, integra com a config atual |
| Manifest | 🟢 Baixa | Cores e nome já definidos |
| Ícones (incl. maskable) | 🟡 Média | Precisa re-exportar logo num canvas quadrado |
| Estratégia de cache (app shell) | 🟢 Baixa | `generateSW` cobre o caso |
| Não quebrar auth por cookie | 🟡 Média | **Ponto crítico**: jamais cachear `/api/` |
| Prompt de atualização (UI) | 🟡 Média | Componente novo + hook `useRegisterSW` |
| Meta tags / splash iOS | 🟡 Média | iOS é mais chato (sem `beforeinstallprompt`) |
| Ajuste nginx | 🟢 Baixa | 1 `location` novo |
| Testes/Lighthouse | 🟢 Baixa | Checklist objetivo |

**Estimativa:** ~1 a 1,5 dia de dev para o MVP instalável bem testado.

### Por que não é "trivial 100%"

Três pontos exigem cuidado consciente (detalhados na §7):

1. **Auth por cookie + SW** — cachear respostas de `/api/` corromperia sessão e
   exibiria dados de outro estado/usuário. A regra é **NetworkOnly** para API.
2. **Dados ao vivo** — jogos/ranking mudam por polling (`refetchInterval`); não
   podem vir de cache de SW.
3. **Atualização do SW** — sem o fluxo de update, o usuário fica preso numa
   versão antiga em cache. Precisa de prompt de reload.

## 4. Decisões de arquitetura

1. **Biblioteca:** `vite-plugin-pwa` (Workbox por baixo). É o padrão da stack
   Vite, gera SW, injeta manifest e oferece `virtual:pwa-register/react`.
2. **Modo de geração:** **`generateSW`** (e não `injectManifest`). O MVP só
   precisa de precache do shell + regras declarativas de runtime caching;
   `generateSW` evita escrever um SW à mão. Migrar para `injectManifest` fica
   para quando entrar push/background sync.
3. **`registerType`: `prompt`** (e **não** `autoUpdate`). Queremos avisar o
   usuário e deixá-lo recarregar — `autoUpdate` pode trocar chunks no meio de um
   palpite. Prompt = controle.
4. **Escopo de precache:** apenas o app shell (`index.html`, JS/CSS de entrada,
   ícones, manifest). As rotas lazy entram via runtime caching sob demanda.
5. **API nunca é cacheada pelo SW.** O cache de dados continua sendo
   responsabilidade do **TanStack Query** (memória), como hoje.

## 5. Plano de implementação (passo a passo)

### Fase 1 — Dependência e configuração do Vite

1. Adicionar dev dependency: `vite-plugin-pwa` (+ `workbox-window` é trazido
   transitivamente).
2. Em `vite.config.ts`, importar e registrar o plugin **depois** de `react()` e
   `tailwindcss()`:

```ts
import { VitePWA } from 'vite-plugin-pwa'

// dentro de plugins: [...]
VitePWA({
  registerType: 'prompt',
  injectRegister: null, // registramos manualmente via React (ver Fase 5)
  includeAssets: [
    'favicon-64.png',
    'bolao_clt_fav.png',
    'apple-touch-icon.png',
  ],
  manifest: { /* ver Fase 2 */ },
  workbox: { /* ver Fase 4 */ },
  devOptions: { enabled: false }, // SW desligado em dev p/ não atrapalhar HMR
})
```

3. Adicionar `vite-plugin-pwa/client` aos tipos. Criar (ou ajustar) um
   `src/vite-env.d.ts` com:

```ts
/// <reference types="vite-plugin-pwa/client" />
/// <reference types="vite-plugin-pwa/react" />
```

> Nota: `tsconfig.app.json` usa `moduleResolution: "bundler"` — as referências
> de tipo acima resolvem o módulo virtual `virtual:pwa-register/react`.

### Fase 2 — Web App Manifest

Configurar o `manifest` no plugin (cores já existem no `index.html`):

```ts
manifest: {
  id: '/',
  name: 'Bolão CLT — Bolão da Copa 2026',
  short_name: 'Bolão CLT',
  description: 'Palpites da Copa do Mundo 2026 com grupos, pontuação e ranking entre amigos.',
  lang: 'pt-BR',
  dir: 'ltr',
  start_url: '/?source=pwa',
  scope: '/',
  display: 'standalone',
  orientation: 'portrait',
  background_color: '#123D2A', // splash bg — alinhar com doc/ui.md
  theme_color: '#123D2A',
  categories: ['sports', 'games', 'social'],
  icons: [ /* ver Fase 3 */ ],
}
```

Pontos de atenção:
- `theme_color`/`background_color` devem bater com os tokens de `doc/ui.md`
  (verde Brasil Essencial `#123D2A`). Confirmar o tom exato no design system.
- `start_url` com `?source=pwa` ajuda a medir aberturas via PWA no GA
  (`src/lib/analytics.ts`).
- `display_override: ['standalone', 'minimal-ui']` é opcional, mas melhora UX.

### Fase 3 — Ícones (precisa de assets novos)

Gerar a partir do logo num **canvas quadrado** com área de segurança (o maskable
exige ~20% de padding seguro). Colocar em `public/icons/`:

| Arquivo | Tamanho | `purpose` | Uso |
| --- | --- | --- | --- |
| `pwa-192.png` | 192×192 | `any` | Android padrão |
| `pwa-512.png` | 512×512 | `any` | Android / splash |
| `pwa-maskable-192.png` | 192×192 | `maskable` | Ícone adaptativo Android |
| `pwa-maskable-512.png` | 512×512 | `maskable` | Ícone adaptativo Android |
| `apple-touch-icon.png` | 180×180 | — (link no HTML) | iOS home screen |
| `favicon.svg` (opcional) | vetor | — | Aba do navegador |

Entrada de `icons` no manifest:

```ts
icons: [
  { src: '/icons/pwa-192.png', sizes: '192x192', type: 'image/png', purpose: 'any' },
  { src: '/icons/pwa-512.png', sizes: '512x512', type: 'image/png', purpose: 'any' },
  { src: '/icons/pwa-maskable-192.png', sizes: '192x192', type: 'image/png', purpose: 'maskable' },
  { src: '/icons/pwa-maskable-512.png', sizes: '512x512', type: 'image/png', purpose: 'maskable' },
]
```

> Geração: usar `@vite-pwa/assets-generator` (CLI `pwa-assets-generator`) a
> partir de um PNG-fonte quadrado de alta resolução (≥512px), ou exportar
> manualmente do logo. **Pré-requisito:** produzir um logo-fonte quadrado, pois
> o `bolao_clt_logo.png` atual é 500×668.

### Fase 4 — Service Worker / estratégias de cache (Workbox)

Configurar `workbox` no plugin. **Regra de ouro: API fora do SW.**

```ts
workbox: {
  // Precache do app shell (build hashed): JS, CSS, HTML, ícones, manifest
  globPatterns: ['**/*.{js,css,html,svg,png,webmanifest}'],
  // SPA fallback: navegação offline cai no index.html...
  navigateFallback: '/index.html',
  // ...EXCETO /api e /admin (admin não deve ser servido offline)
  navigateFallbackDenylist: [/^\/api\//, /^\/admin\//],
  cleanupOutdatedCaches: true,
  clientsClaim: true,
  skipWaiting: false, // par com registerType:'prompt' (update controlado)

  runtimeCaching: [
    // 1) API: NUNCA cachear (sessão por cookie + dados ao vivo)
    {
      urlPattern: ({ url }) => url.pathname.startsWith('/api/'),
      handler: 'NetworkOnly',
    },
    // 2) Bandeiras/escudos e imagens remotas: StaleWhileRevalidate
    {
      urlPattern: ({ request }) => request.destination === 'image',
      handler: 'StaleWhileRevalidate',
      options: {
        cacheName: 'img-cache',
        expiration: { maxEntries: 120, maxAgeSeconds: 60 * 60 * 24 * 30 },
      },
    },
    // 3) Fontes Google (se houver): CacheFirst
    {
      urlPattern: ({ url }) => url.origin === 'https://fonts.gstatic.com',
      handler: 'CacheFirst',
      options: {
        cacheName: 'font-cache',
        expiration: { maxEntries: 20, maxAgeSeconds: 60 * 60 * 24 * 365 },
      },
    },
  ],
}
```

Pontos de atenção:
- **`NetworkOnly` para `/api/`** é o que protege a auth por cookie e os dados de
  jogos ao vivo (`useMatchLive`, `useGroupLiveMyPoints` etc.).
- Não cachear o `gtag.js` (GA) — ele já é carregado dinamicamente só após
  consentimento (`src/lib/analytics.ts`, `src/lib/cookieConsent.ts`); manter
  fora do SW para não burlar o opt-in da LGPD.
- `navigateFallbackDenylist` impede que `/admin/*` seja servido pelo shell
  offline (área sensível, login separado — ver `doc/010-authentication.md`).

### Fase 5 — Registro do SW + prompt de atualização (UI)

1. Criar um hook/wrapper de PWA, ex. `src/lib/pwa.ts` ou componente
   `src/components/pwa/PwaUpdatePrompt.tsx`, usando o módulo virtual:

```tsx
import { useRegisterSW } from 'virtual:pwa-register/react'

export function PwaUpdatePrompt() {
  const {
    needRefresh: [needRefresh, setNeedRefresh],
    updateServiceWorker,
  } = useRegisterSW({
    onRegisteredSW(_swUrl, reg) {
      // checa update periodicamente (ex.: a cada 60min)
      reg && setInterval(() => reg.update(), 60 * 60 * 1000)
    },
  })

  if (!needRefresh) return null
  return (
    /* Toast/banner usando o ToastProvider (src/context/toast.context)
       ou um banner próprio seguindo doc/ui.md */
    <UpdateBanner
      onReload={() => updateServiceWorker(true)}
      onDismiss={() => setNeedRefresh(false)}
    />
  )
}
```

2. Montar o componente no `src/main.tsx`, ao lado de `<CookieConsent />`, dentro
   do `ToastProvider`. Reaproveitar o padrão visual do toast existente para a
   mensagem "Nova versão disponível — Atualizar".

3. (Opcional) Capturar `beforeinstallprompt` para um **botão "Instalar app"** no
   perfil (`src/pages/profile/ProfilePage`) ou no `GroupShell`. No iOS não há
   esse evento → mostrar instrução "Compartilhar → Adicionar à Tela de Início".

### Fase 6 — Meta tags iOS no `index.html`

O iOS ignora boa parte do manifest; adicionar no `<head>`:

```html
<link rel="apple-touch-icon" href="/icons/apple-touch-icon.png" />
<meta name="apple-mobile-web-app-capable" content="yes" />
<meta name="apple-mobile-web-app-status-bar-style" content="black-translucent" />
<meta name="apple-mobile-web-app-title" content="Bolão CLT" />
<meta name="mobile-web-app-capable" content="yes" />
```

(Splash screens iOS são opcionais e numerosas; o `@vite-pwa/assets-generator`
pode emiti-las, mas dá para deixar para uma iteração posterior.)

### Fase 7 — nginx (headers de cache)

No `nginx/default.conf.template`, **antes** do `location /`, garantir que o SW e
o manifest **não** fiquem presos em cache de navegador/CDN:

```nginx
location = /sw.js {
  add_header Cache-Control "no-cache, no-store, must-revalidate";
  expires off;
}

location = /manifest.webmanifest {
  add_header Cache-Control "no-cache";
  types { } default_type application/manifest+json;
}

location = /registerSW.js {
  add_header Cache-Control "no-cache, no-store, must-revalidate";
}
```

- `/assets/` continua `immutable` (são hashados pelo Vite) — **não** mexer.
- Adicionar `application/manifest+json` ao `gzip_types` se quiser comprimir o
  manifest (opcional).
- O nome real do arquivo do SW gerado pelo `vite-plugin-pwa` em modo
  `generateSW` é `sw.js` por padrão — confirmar após o primeiro build e ajustar
  o `location` se necessário.

### Fase 8 — Build, Docker e validação

- O `Dockerfile` **não muda**: `bun run build` já vai emitir `sw.js`,
  `manifest.webmanifest` e ícones em `dist/`, copiados para o nginx.
- Confirmar que `public/icons/*` entram no build (Vite copia `public/` para a
  raiz de `dist/`).
- Rodar `bun run typecheck` e `bun run build` localmente; servir o `dist` por
  HTTPS (ou `localhost`, que conta como secure origin) para testar o SW.

## 6. Matriz de cache (resumo)

| Recurso | Origem | Estratégia SW | Justificativa |
| --- | --- | --- | --- |
| `index.html`, JS/CSS de entrada | build | Precache | App shell instantâneo |
| Chunks de rota (lazy) | build | Precache (hashados) | Já são imutáveis |
| Ícones / manifest | build | Precache | Necessários p/ instalar |
| **`/api/*`** | backend | **NetworkOnly** | **Cookie de sessão + dados ao vivo** |
| `/admin/*` (navegação) | SPA | Sem fallback offline | Área sensível |
| Imagens (escudos/bandeiras) | remoto/local | StaleWhileRevalidate | UX rápida, atualiza em background |
| Fontes | Google Fonts | CacheFirst | Estáveis e versionadas |
| `gtag.js` (GA) | Google | Fora do SW | Respeitar consentimento LGPD |

## 7. Riscos e pontos de atenção

1. **Auth por cookie httpOnly + SW** *(crítico)*: cachear `/api/` mostraria dados
   do estado errado ou de outra sessão. Mitigação: `NetworkOnly` em `/api/` +
   `navigateFallbackDenylist`. Já tratado na Fase 4.
2. **Dados ao vivo**: `useMatchLive` (polling 60s), `useGroupLiveMyPoints`,
   ranking ao vivo — todos dependem de rede fresca. Não cachear API garante isso.
3. **Versão presa em cache (stale)**: sem o fluxo de update o usuário não recebe
   correções. Mitigação: `registerType: 'prompt'` + `PwaUpdatePrompt` +
   `reg.update()` periódico. Fase 5.
4. **iOS**: sem `beforeinstallprompt`, sem prompt automático, suporte parcial a
   manifest. Mitigação: meta tags da Fase 6 + instrução manual de instalação.
5. **Troca de chunk durante uso**: por isso `skipWaiting: false` —
   o SW novo só assume após o usuário aceitar recarregar.
6. **LGPD / GA**: manter `gtag.js` fora do precache e do runtime caching para não
   carregar analytics sem opt-in (`src/lib/cookieConsent.ts`).
7. **Dev/HMR**: `devOptions.enabled: false` evita que o SW interfira no
   hot-reload do `bun run dev`.
8. **HTTPS obrigatório**: SW só funciona em origem segura. Produção
   (`https://bolaoclt.com.br`) ok; em testes usar `localhost`.

## 8. Critérios de aceite

- [ ] Lighthouse → categoria **PWA: Installable** ✅ e sem erros de manifest/SW.
- [ ] Botão/prompt **"Instalar"** aparece no Android (Chrome) e a instalação
      coloca o ícone correto (não-borrado, com maskable) na tela inicial.

      `background_color`/ícone.

      de `/api/` é servido do cache.

      e ao confirmar a app recarrega na versão nova.

      `Bolão CLT`; abre em tela cheia.
